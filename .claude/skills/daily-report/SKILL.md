# 日报生成 Skill

从 `00_Inbox` 和 `01_Daily/logs` 中读取临时记录，按日报模板整理成结构化日报。

## 触发条件

用户说以下任意内容时触发：
- "生成日报" / "汇总日报" / "整理日报" / "写日报"
- "今日日报" / "昨日日报" / "昨天的日报"
- "汇总下XX日的记录" / "把XX日的内容整理成日报"

## 执行流程

### 第1步：确定日期

- 用户明确指定日期 → 使用该日期
- 用户说"今日"/"今天" → 使用当天日期
- 用户说"昨日"/"昨天"或未指定 → 使用昨天日期
- 日期格式统一为 `YYYY-MM-DD`

### 第2步：读取数据源

并行读取以下文件（文件不存在则跳过）：

1. **Inbox 临时文档**：`00_Inbox/YYYY-MM-DD.md`
   - 这是主要数据源，包含当日零散的工作记录
2. **随手记**：`01_Daily/logs/YYYY-MM-DD.md`
   - 辅助数据源，可能包含补充信息
3. **日报模板**：`01_Daily/Templates/日报模板.md`
   - 日报的格式规范

如果两个数据源都不存在，告知用户"未找到该日期的记录"并结束。

### 第3步：读取计划总览（可选）

读取 `01_Daily/计划总览.md`，如果其中有与当日相关的进行中任务或已完成任务，可纳入日报。

### 第4步：整理日报内容

将原始记录按以下规则整理：

**分类归组**：将零散记录按项目/领域分组，如：
- 前端/引擎相关
- 服务器运维
- 某项目名称（如"传奇项目"）
- 会议/沟通
- 学习/调研

**提取待跟进事项**：识别记录中带有"待做"、"需要"、"待测试"、"待审核"等关键词的条目，单独提取为待跟进 checklist（`- [ ]` 格式）。

**时间线保留**：保留原始记录中的时间节点信息，但不要逐条罗列，而是归纳到各分类下。

### 第5步：生成日报文件

按模板格式生成日报，保存到：
```
01_Daily/daily/YYYY-MM/YYYY-MM-DD.md
```

- 如果 `YYYY-MM` 目录不存在，先创建
- 文件名就是日期，不加其他前缀

日报结构示例：
```markdown
# YYYY-MM-DD 日报

> 记录时间：HH:MM - HH:MM

---

## 今日完成

### 分类一
- 事项描述

### 分类二
- 事项描述

---

## 待跟进

- [ ] 待办事项1
- [ ] 待办事项2

---

## 备注

补充说明（如有）
```

### 第6步：同步待跟进到计划总览

日报中的待跟进事项，同时同步到 `01_Daily/计划总览.md` 的"进行中"区域。

**同步规则**：
- 每条待跟进事项追加到计划总览的"进行中"列表末尾
- 格式：`- [ ] 事项描述（来源：YYYY-MM-DD 日报）`
- 如果计划总览中已存在相同事项（模糊匹配），不重复添加
- 保持计划总览中其他已有内容不变

**读取计划总览时**：需要先读取当前内容，判断哪些事项已存在，再决定追加哪些。

**日报中待跟进事项的格式**：保留在日报中作为历史记录，不做改动。

### 第7步：标记原始文档已处理

在原始 Inbox 文档的 YAML frontmatter 中添加 `status: 已处理` 标签：

- 如果文件已有 frontmatter（以 `---` 开头），在其中添加 `status: 已处理`
- 如果文件没有 frontmatter，在文件开头添加：
```yaml
---
status: 已处理
---
```

随手记文件不做标记处理（它本身就是原始记录）。

### 第8步：汇报结果

生成完成后，告知用户：
1. 日报保存路径
2. 原始文档是否已标记
3. 待跟进事项数量及同步到计划总览的情况

## 注意事项

- 忠于原文：整理时不能歪曲原始记录的含义
- 按照项目 `CLAUDE.md` 中的命名规范执行（短横线命名法、日期格式等）
- 日报中的分类名称应简洁明了，与项目实际工作领域对应
- 如果原始记录非常简短（只有1-2条），日报也保持简洁，不要注水
