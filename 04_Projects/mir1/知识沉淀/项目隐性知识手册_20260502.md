**生成日期**: 2026-05-02

**定位**: 记录那些无法从文件名/类名直接推导、需要通读代码才能理解的隐性约定和关键知识。供新成员快速理解项目设计决策，也供后续审核/重构时参考。

## 1. 并发模型与锁约定

### 1.1 三层并发保护机制

本项目使用了三种不同的并发保护机制，**混用且无统一规范**，是理解并发行为的关键：

| 机制 | 实现方式 | 保护粒度 | 使用场景 | 风险 |
|------|----------|----------|----------|------|
| `@AsyncBindExecute` | 注解+玩家消息队列 | 玩家级串行化 | SkillService, RoleEnterGameRunner等 | 不保护非消息线程的调用(如定时任务、异步回调) |
| `ReentrantLock` | BagEntity.lock / fightModelEntity.lock | 实体级互斥 | BagService, BuffManager, FightModelEntity | tryLock超时后行为不一致(有的return有的继续执行) |
| `synchronized` | 方法级/类级/对象级 | 取决于锁对象 | ShopService.addAuction, GuildService.createGuild, TransactionService.createOrder | 锁粒度不统一(类锁vs实例锁vs订单锁) |

### 1.2 关锁位置速查

```
BagEntity.lock                → BagService (10+处 tryLock 5秒)
                                addItem → changeCell (重入)
                                
fightModelEntity.lock         → BuffManager.addBuff/removeBuff (tryLock 2-5秒)
                                ⚠️ checkEnterBuff 未获取此锁!
                                
synchronized(ShopService)     → addAuction (Center, 方法级)
                                ⚠️ buyRmbShop 用 @Locked AOP(实例锁), 不等效

synchronized(TransactionService.class) → createOrder (类级锁, 太粗)
synchronized(orderModel)      → changeTransaction (订单级锁)
                                ⚠️ lockTransaction 无任何同步!

synchronized(GuildService)    → 仅 createGuild
                                ⚠️ dealRequestList/transferMaster/quitGuild/kickMember 均无锁
```

### 1.3 消息线程 vs 业务线程

理解哪些代码在哪个线程执行，是分析竞态条件的前提：

| 执行线程 | 代码位置 | 特点 |
|----------|----------|------|
| **消息处理线程** (QueueMessageProcessor) | Handler.execute() → Service方法 | 按 `aid.hashCode()%N` 绑定到固定线程，同aid消息串行 |
| **地图绑定线程** (BindThreadPoolManager) | BattleProcessor.run(), RemoveModelTask | 同mapId任务在固定线程，但跨地图可并发 |
| **定时调度线程** (ScheduledThreadPoolManager) | Scheduled方法, 延迟任务 | 独立线程，与消息线程并发 |
| **Spring事件线程** (EventListener) | TaskDispatcher, GuildService事件 | 同步事件在发布者线程执行，异步事件在独立线程 |
| **异步登录线程** (ThreadPoolManager) | RoleLoginRunner.login() | 无@AsyncBindExecute，可与其他线程并发操作同一gamePlayer |
| **异步邮件线程** (ScheduledThreadPoolManager) | MailService.doReceiveMails() | CAS防重入，但内部requestMsg可超时 |
| **拍卖处理线程** (AuctionNewProcessor) | handleBuy/handleDown等 | 单线程队列消费，保证拍卖操作串行 |

### 1.4 tryLock超时行为不一致

项目中 `ReentrantLock.tryLock(timeout)` 超时后的处理方式**没有统一约定**：

| 调用位置 | 超时行为 | 是否安全 |
|----------|----------|----------|
| BagService.addItem | 超时返回false，外层检查lockFlag | ✅ 安全 |
| BuffManager.refreshBagWeightBuff | 超时仅log，**继续执行后续buff操作** | ❌ 危险 |
| SkillService.skillToggle→changeProp | changeProp超时返回false，**但skillToggle不检查返回值** | ❌ 危险 |
| FightCheckService.checkAttackSkills | changeProp超时返回false，**不检查返回值** | ❌ 危险 |

**约定建议**: 所有tryLock超时后必须 `return` 或抛异常，禁止"打日志后继续执行"。

## 2. 消息路由与Handler注册机制

### 2.1 注册流程

```
启动时扫描:
  Classpath → Scannotation → 扫描 @Handler 注解的类
    → HandlerManager.registerHandler()
        → 根据 getTypes() 返回的 msgCode 数组注册映射

运行时路由:
  MessageProcessorDispatcher.put(msg)
    → Math.abs(msg.getAid().hashCode()) % processors.size()  ← ⚠️ Integer.MIN_VALUE问题
    → QueueMessageProcessor 消费
    → HandlerManager.execute(msg)
        → handlers.get(msg.getType())  ← HashMap查找
        → handler.execute(msg)
```

### 2.2 Handler→Service映射约定

所有Handler遵循统一模式：**Handler只做消息解析和路由，不含业务逻辑**。

```java
@Handler
@Component
public class XXXHandler implements MessageHandler {
    @Override
    public void execute(Msg msg) {
        XXXService service = SpringContextUtil.getBean(XXXService.class);
        switch (msg.getType()) {
            case GameMsgCode.CS_xxx_msg_code:
                XxxMsg.CS_xxx_msg csMsg = (XxxMsg.CS_xxx_msg) msg.getData();
                service.xxxMethod(uid, csMsg.getParam1(), csMsg.getParam2());
                break;
        }
    }
    
    @Override
    public int[] getTypes() {
        return new int[]{ GameMsgCode.CS_xxx_msg_code };
    }
}
```

### 2.3 中心服特殊路由

中心服使用**通用消息+type字段**做二次分发，不直接走Handler：

```
Game → Center:
  CG_send_general_msg(type, Map<String,String> datas)
    → Game2CenterHandler → 根据 type 分发到 CenterToGameMsgBizService 实现
  
  CG_send_sync_general_msg(type, Map<String,String> datas)  ← 有返回值
    → Game2CenterHandler → 根据 type 分发到 CenterToGameSyncMsgBizService 实现

Center → Game:
  CG_send_general_msg(type, datas) → Center2GameHandler → type分发

type值定义: CommonConstants.CenterToGameMsgType / CenterToGameSyncMsgType
```

**关键约定**: Center→Game的同步调用(`requestMsg`)有超时机制，超时返回null。各业务方对null的处理不一致（有的当失败，有的当成功），是重要风险来源。

## 3. 实体类生命周期与持久化约定

### 3.1 实体类继承体系

```
BaseEntity (抽象基类)
    ├─ BagEntity          ← 背包(含lock, cells, virtualItemMap)
    ├─ RoleEntity         ← 角色基础数据
    ├─ RoleFightModelEntity ← 战斗属性(含lock, buffMap, propMap)
    ├─ MapEntity          ← 地图数据
    ├─ MailEntity         ← 邮件
    ├─ GuildEntity        ← 公会
    ├─ AuctionNewEntity   ← 拍卖
    ├─ TaskEntity         ← 任务
    └─ ...

关键约定:
  - 所有Entity通过 FastJSON 序列化存入Redis
  - @JSONField(serialize=false) 标记非持久化字段(如lock对象)
  - Entity通过 DataHelper.getInstance().save(entity) 持久化
  - Entity通过 DataHelper.getInstance().getEntity(Class, rid) 加载
```

### 3.2 save()语义 — 全量覆盖写入

**关键隐性知识**: `DataHelper.save(entity)` 是**全量覆盖写入Redis**，不是增量更新。

```
save(entity):
  1. FastJSON.toJSONString(entity)  ← 序列化整个实体
  2. redisService.set(key, json)    ← 整体覆盖

这意味着:
  ✅ 优点: 简单，不存在部分更新的一致性问题
  ❌ 缺点: 每次changeCell都触发全量save，大Entity(如BagEntity)序列化开销大
  ❌ 缺点: 高频save下，两次save之间如果并发修改，后写的覆盖先写的
```

### 3.3 Entity的获取路径

```
方式1: 从内存Map获取(在线玩家)
  DataHelper.getInstance().getGamePlayer(aid).getEntity(XxxEntity.class)
  → gamePlayer.entityMap.get(XxxEntity.class)

方式2: 从Redis获取(离线玩家)
  DataHelper.getEntity(XxxEntity.class, rid)
  → redisService.get(key) → FastJSON.parse(json)

方式3: 跨服获取
  CenterGameUtil → GameToGameMessageManager.requestMsg()
  → 同步RPC到目标服 → 目标服加载Entity → 序列化返回
```

**⚠️ 关键约定**: `gamePlayer.getEntity()` 在entityMap为空时返回null，不会自动从Redis加载。跨服跳转后`clearEntity()`清空了entityMap，此时getEntity返回null。

### 3.4 实体脏标记与刷盘

```
BagService.markBagDirty():
  bagEntity.setDirty(true)
  bagEntity.setLastDirtyTime(now)
  bagEntity.setDirtyRevision(dirtyRevision + 1)
  ⚠️ 三步操作非原子

executeBagFlushTask() (异步):
  if(dirty && now - lastDirtyTime > interval) {
    savingFlag = true
    save(bagEntity)  ← 全量save
    dirty = false
    savingFlag = false
  }

约定: BagEntity 不是每次changeCell都save，而是异步定时刷盘。
但某些关键路径(如交易确认)会直接调用save()强制持久化。
```

## 4. 跨服通信协议约定

### 4.1 Game↔Center消息封装

```java
// Game → Center 通用消息(无返回)
GameToCenterUtil.publishToCenter(type, datas)
  → Nats publish → Center Game2CenterHandler

// Game → Center 同步消息(有返回)
GameToCenterUtil.requestToCenter(type, datas)
  → Nats request → Center Game2CenterHandler → 返回Msg
  → ⚠️ 超时返回null，各业务方处理不一致

// Center → Game 通用消息
centerMsgService.convertCG_send_general_msg(type, datas)
  → Nats publish → Game Center2GameHandler

// Center → Game 同步消息(有返回)
centerMsgService.convertCG_send_sync_general_msg(type, datas)
  → Nats request → Game Center2GameHandler → 返回Msg
```

### 4.2 datas参数约定

所有跨服通用消息的 `datas` 是 `Map<String, String>`，通过 `CommonConstants` 中定义的Key传递参数：

| Key常量 | 用途 |
|---------|------|
| MSG_ROLE_RID | 角色rid |
| MSG_ITEMS | 物品列表(JSON序列化) |
| MSG_ITEM_CHANGE_TYPE | 物品变更原因(LogConstants.ItemChangeType) |
| MSG_SOURCE_UNIQUE | 幂等唯一值(防重复添加物品) |
| MSG_SECT_KEY | 公会ID |
| MSG_SECT_NAME | 公会名称 |
| MSG_MAP_ACTION_TYPE | 地图动作类型 |
| MSG_COMMON_VALUE1/2/3 | 通用数值参数 |

**⚠️ 关键约定**: `MSG_SOURCE_UNIQUE` 是物品添加的幂等键，格式为业务前缀+业务ID（如邮件:`bag_unique_pre+mailId`）。Game服添加物品时会检查此唯一值，防止重复添加。

### 4.3 跨服数据一致性模型

```
弱一致性模型:
  - Game服和Center服各自维护内存中的Entity副本
  - 通过save()写入Redis作为共享状态
  - 修改后需要主动通知对方刷新

  Game服修改 → save(Redis) → publish通知Center → Center重新从Redis加载
  Center修改 → save(Redis) → publish通知Game → Game重新从Redis加载

  ⚠️ 如果通知丢失或延迟，两服数据不一致
  ⚠️ 没有版本号/时间戳机制检测冲突
```

## 5. 经济操作原子性约定

### 5.1 扣减-发放操作模式

项目中所有经济操作遵循 `先扣后发` 模式，但**没有事务保障**：

```
模式A: 直接扣减+发放(无锁)
  if(reduceBagItem(player, cost)) {    ← 扣款
      addItem(player, reward);          ← 发放
  }
  风险: reduce成功但addItem失败 → 物品丢失，无回滚

模式B: 检查+扣减+发放(无锁check)
  if(getBalance() >= cost) {            ← 检查(无锁)
      reduceBagItem(player, cost);      ← 扣款(有锁)
      addItem(player, reward);          ← 发放
  }
  风险: check与deduct之间余额可变 → 负余额

模式C: 检查+扣减+发放(有锁，正确)
  lock();
  if(getBalance() >= cost) {            ← 检查(有锁)
      reduceBagItem(player, cost);      ← 扣款(有锁内)
      addItem(player, reward);          ← 发放
  }
  unlock();
  ✅ 安全
```

**当前项目大部分经济操作使用模式A或B**，模式C仅出现在少数有`@Locked`注解的方法。

### 5.2 物品添加的幂等保护

通过 `MSG_SOURCE_UNIQUE` 实现跨服物品添加的幂等：

```java
// Center → Game 添加物品时携带唯一值
datas.put(CommonConstants.MSG_SOURCE_UNIQUE, "bag_unique_mailId123");

// Game服 addItem 内部:
if(redisService.hasKey(uniqueKey)) {
    return;  // 已添加过，幂等返回
}
// 正常添加物品
redisService.set(uniqueKey, "1");  // 标记已添加
```

**⚠️ 但**: 同一进程内的 `addItem` 调用**没有幂等保护**，仅跨服同步调用有。

### 5.3 货币类型体系

```
虚拟物品(通过BagEntity.virtualItemMap管理):
  - 金币(ItemConfig.id对应)
  - 绑定金币
  - 元宝
  - 绑定元宝
  - 其他代币

  读取: bagEntity.getVirtualItemMap().getOrDefault(itemId, 0L)
  扣减: bagService.reduceBagItem(gamePlayer, Item.create(itemId, count))
  ⚠️ virtualItemMap 的 get+put 操作非原子(getOrDefault只保证读)

实体物品(通过BagEntity.bagCells管理):
  - Cell列表，每个Cell包含Item
  - 可堆叠/不可堆叠物品区分(canmerge字段)
  - 扣减时区分: reduceBagItem(按itemId) vs reduceBagItemWithTypeIndex(按格子位置)
```

## 6. 配置表与代码的桥接约定

### 6.1 ConfigService加载机制

```
ConfigService.getInstance().get(ItemConfig.class, id)
  → 从内存Map中查找
  → 启动时通过 @Excel 注解的类从Excel配置表加载
```

### 6.2 @Excel注解映射

配置实体类通过 `@Excel` 注解映射Excel列到Java字段：

```java
@Excel(name = "ItemConfig")    // Excel表名
public class ItemConfig {
    @ExcelProperty("id")       // 列名映射
    private int id;
    @ExcelProperty("price")
    private int price;
    // ...
}
```

**⚠️ 关键约定**: `ItemConfig.getPrice()` 返回int，与int类型的count相乘会溢出。这是一个**配置读取侧的类型陷阱**，多处代码踩坑(ShopService:163, 524)。

### 6.3 配置热更

```
Center服:
  ExcelController → 上传新Excel → ConfigService.reload()
  → 通知Game服刷新配置

Game服:
  ExcelController → 上传新Excel → ConfigService.reload()

⚠️ 配置热更期间:
  - 正在执行的Service方法可能读到新旧配置的混合状态
  - 配置对象本身无版本号，无法检测stale read
  - monsterConfig等对象被synchronized锁住时可能影响其他逻辑
```

## 7. 枚举路由陷阱

### 7.1 ordinal路由机制

```
GameMsgCode 定义消息码(非enum，是interface中的int常量):
  CS_shop_buy_msg_code = 10001
  
业务枚举使用 ordinal() 路由:
  GuildPosition.values()[ordinal]  → 职位
  AuctionState.values()[ordinal]   → 拍卖状态
  MailState.values()[ordinal]      → 邮件状态

⚠️ CLAUDE.md已记录: 在枚举中间插入成员会导致ordinal偏移
⚠️ 补充: 枚举的 getInstance(int value) 方法通过遍历匹配ordinal:
  public static GuildPosition getInstance(int value) {
      for(GuildPosition e : values()) {
          if(e.ordinal() == value) return e;
      }
      return null;
  }
  如果传入越界value → 返回null → 大部分代码不检查null → NPE
```

### 7.2 GameMsgCode分段约定

```
消息码分段:
  10001-19999   Game服消息(CS_xxx → GameHandler)
  20001-22500   Game → Center消息
  22501-25000   Center → Game消息
  25001-30000   跨服消息

⚠️ 消息码是interface中的int常量，不是enum，不会因插入而偏移
⚠️ 但消息码间有空隙(预留位)，新消息应追加到各段末尾
```

## 8. 关键Anti-Pattern记录

以下模式在项目中反复出现，是理解bug根因的关键：

### 8.1 check-then-act无锁

**出现次数**: 10+处
**典型代码**:
```java
if(getBalance() >= cost) {  // check: 无锁读取
    reduce(cost);            // act: 有锁扣减
    add(reward);             // act: 有锁发放
}
```
**根因**: 项目早期单线程架构遗留，后引入多线程消息处理但未同步修复check路径。

### 8.2 扣减成功→发放失败→无回滚

**出现次数**: 5+处（交易、任务、商店）
**典型代码**:
```java
if(reduceBagItem(player, cost)) {
    boolean b = addItem(player, reward);
    if(!b) { log.error("添加物品失败"); }  // 仅日志，不回滚
}
```

### 8.3 synchronized锁粒度混乱

**同模块内三种锁粒度**:
- `synchronized(TransactionService.class)` — 类锁(createOrder)
- `synchronized(orderModel)` — 订单锁(changeTransaction)  
- 无锁(lockTransaction)

**原因**: 不同时期不同开发者添加，无锁规范文档。

### 8.4 客户端参数直接使用

**高危参数清单**:

| 参数 | 来源 | 风险 | 正确做法 |
|------|------|------|----------|
| `count` | CS_shop_buy_msg | 溢出/负数 | 服务端校验范围 |
| `perPrice` | CS_auction_add_msg | 绕过限价 | 服务端从配置读取 |
| `currencyType` | CS_auction_add_msg | 绕过货币类型限制 | 服务端从配置读取 |
| `castItemId` | CS_shop_buy_rmb_msg | 替换消耗物品 | 服务端从配置读取 |
| `money` | 充值回调 | 伪造金额 | 服务端与配置比对 |
| `selectIndex` | CS_task_complete_msg | 越界取奖励 | 校验 < rewardList.size() |

## 9. 线程池与调度约定

### 9.1 线程池清单

| 线程池 | 管理类 | 用途 | 线程数 |
|--------|--------|------|--------|
| 消息处理线程池 | MessageProcessorDispatcher | 消息消费 | CPU*2 |
| 地图绑定线程池 | BindThreadPoolManager | 同一地图任务串行 | 按地图数 |
| 定时调度线程池 | ScheduledThreadPoolManager | 延迟任务/定时任务 | 配置 |
| 通用异步线程池 | ThreadPoolManager | 登录等异步操作 | 配置 |
| 战斗帧驱动线程 | BattleProcessorManager | tick驱动 | 1(单线程调度) |
| 拍卖处理线程 | AuctionNewProcessor | 拍卖操作串行 | 1 |

### 9.2 定时任务执行约定

```
@Scheduled 注解:
  - 需 @EnableScheduling 启用
  - 默认单线程执行，长任务会阻塞其他定时任务
  - 项目中使用 ScheduledThreadPoolManager 提交延迟任务以避免阻塞

BindThreadPoolManager:
  - 每个mapId绑定到一个固定线程
  - 同一地图的任务保证串行(但不同地图可并发)
  - ⚠️ 地图数多时线程爆炸(每个地图一个线程)
```

## 10. 关键类职责速查

| 类名 | 真实职责(与类名可能不符) | 关键隐性知识 |
|------|------------------------|-------------|
| `DataHelper` | 全局状态桶+数据路由+工具方法 | 1275行God Class，单例缺volatile，是所有并发问题的交汇点 |
| `BagService` | 背包操作+仓库+物品使用+定时刷盘 | tryLock 5秒阻塞消息线程，withBagLock已有但多数方法未用 |
| `BattleProcessor` | 帧驱动+实体管理+NPC逻辑+消息转发 | 6个ConcurrentHashMap按类型分存，dropMap的value是普通List |
| `GameToCenterUtil` | Game→Center消息封装 | publish是异步，request是同步(可超时返回null) |
| `CenterGameUtil` | Center→Game物品操作封装 | reduceBagItem/addItem走同步RPC，超时返回false |
| `TransactionStateHelper` | 交易状态Redis标记 | 防止交易中同时进行其他物品操作，但非强制 |
| `PlayerOfflineCacheManager` | 离线缓存+定时清理 | 版本号机制防误清理，但cancel+schedule非原子 |
| `HandlerManager` | 消息路由注册表 | 3个static HashMap，启动写+运行读，HashMap不安全但当前只读场景风险低 |
| `MessageProcessorDispatcher` | 消息哈希分发 | stopLatch为null+final→stop()必NPE(运行中不影响) |
| `AuctionNewProcessor` | 拍卖队列消费 | 继承QueueMessageProcessor，单线程保证拍卖操作串行 |
