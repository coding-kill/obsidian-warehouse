## 4.2 用户内存缓存结构

### UserManager 核心结构

| 缓存集合 | 含义 | 说明 |
|---------|------|------|
| m_usingList | 在线用户 | 完整持有用户对象（背包/货币/状态/任务） |
| m_waitList | 等待态用户 | 断线重连窗口、短时保留 |
| m_accListByAccountId | 账号→User索引 | 快速账号查找 |
| m_listByFamilyName | 家族名→User索引 | 家族相关查询 |
| m_freeList | User对象池 | 复用对象，减少分配 |

### 离线缓存处理流程

```
断线发生
    │
    ▼
moveWaitUser() ──► 从using移入waitList
    │
    ▼
等待期内（可接收globalSend/globalCurrencyAdd/globalItemAdd）
    │
    ├──► 重连成功 ──► onReconnect() ──► 恢复到ONLINE
    │
    └──► 超时(TICK_RECONNECT_USER_WAIT) ──► userReturn() ──► 回对象池
```

### 一致性保证原则

| 原则 | 说明 |
|------|------|
| 单主写 | 同一时刻只有一个服"主写"该用户内存态 |
| 短离线不丢态 | WAIT保留内存对象，允许快速重连 |
| 跨服前先收口 | MOVE_OUT前做saveUser + 场景移除 |
| 最终落库兜底 | DBAgent异步落地保证最终一致 |

## 4.4 切服状态限制

**收到MoveServer后，用户状态变为MOVE_OUT_ZONE，此时**：
- 普通业务命令被拒绝（战斗/道具/商店等）
- 只做必要的迁服收尾回包
- 通过 `sendAfterRecvPacketProcess()` 做最终回包
- 等客户端连上新服后恢复正常处理

## 4.5 排查用户问题盯的4个断点

| 断点 | 含义 | 排查方向 |
|------|------|---------|
| `sNetRecv_UserConnect` | 接管是否成功 | 新服是否收到连接 |
| `moveWaitUser` | 异常断开是否正确入wait | 是否误入对象池 |
| `sNetRecv_MoveServer` | 跨服前是否save+remove | 数据是否丢失 |
| `checkWaitList` | 是否误超时回收 | "刚断就丢"问题 |
